The users can follow the info in this repo to build their own schema for their experiment üëç. Further explanations on NOMAD are availbale in the official documentation page Each folder contains several kind of files, the user can try to drag and drop all of them in the upload page in Nomad, they will be automatically parsed to create "Entries" containing the experimental data in a structured fashion: schema file: it is a NOMAD archive file (.archive.yaml extension) containing only one section called "definitions". It defines the structure that will host your data. Sections, subsections and quantities are the elements that compose the hierarchical structure of data. test datasets: depending on the experiment, the user will have one or more files where the logged parameters and metadata are stored. This guide explains how to develop a custom normalizer in NOMAD using the plugin system. Normalizers enrich parsed data by adding or modifying content within the archive, such as computing additional properties or standardizing values. The document covers how to define a NormalizerEntryPoint, implement a Normalizer class (or SystemBasedNormalizer for system-specific logic), register the normalizer in the plugin configuration, control execution order via levels, and run the normalizer via the CLI or within Python scripts.

This guide explains how to create example uploads in NOMAD using the plugin system. Example uploads provide users with ready-to-use datasets for demonstration, education, or testing purposes. The guide covers how to define an ExampleUploadEntryPoint, how to structure and register the upload, and different ways to include datasuch as embedding it in the plugin, downloading from external URLs, or dynamically generating it with custom code. Example uploads appear in the NOMAD GUI and can be instantiated by users with a single click.

This guide provides comprehensive instructions on creating schema packages in NOMAD using the plugin system. It covers defining custom metainfo schemas in Python, registering them through plugin entry points, and using schema packages to structure and validate scientific data. Youll learn how to organize quantities and sections, extend existing NOMAD schemas, implement normalization logic, version schemas safely, and even migrate old data. This is essential for developers and Oasis administrators aiming to define domain-specific models, support ELN usage, or enhance parsing workflows with structured metadata.

This guide introduces the plugin system of NOMAD, which allows administrators and developers to customize NOMAD installations by adding functionality through Python packages. It explains the anatomy of a plugin, including entry points for parsers, apps, normalizers, APIs, and schema packages. The guide provides recommendations for naming conventions, testing with pytest, formatting with ruff, and documenting with mkdocs. It also outlines how to publish plugins via GitHub or PyPI and how to control plugin activation through the nomad.yaml configuration file.

This guide explains how to write a custom parser in NOMAD using the plugin system. It walks you through creating a ParserEntryPoint, implementing a Parser or MatchingParser class, defining file matching rules, and populating the NOMAD archive with extracted data. The document also covers best practices for parsing text files using regular expressions, working with the NOMAD metainfo system, and integrating schema extensions. Both command-line and Python-based parser execution are demonstrated, enabling developers to test and deploy custom data ingestion workflows.

This guide explains how to develop a custom app in NOMAD using the plugin system. Apps enable domain-specific user interfaces in the NOMAD GUI by defining tailored search filters, result displays, and dashboards. Youll learn how to define an AppEntryPoint, configure the app layout using the App class, include schema-defined quantities, create dashboards and menus, and filter displayed entries. This customization is essential for highlighting domain-relevant data and making the NOMAD GUI intuitive for specific communities or workflows.
